Reading, watching, listening — recorded as memory

Scroll, dwell on the images, and play the video. Every tracked element below streams an engagement event to the server (watch the live log on the right). Each becomes a channel:web · direction:exposure entry in the customer's awareness store.

What gets tracked

Text blocks marked with data-wb-text are tracked for genuine reading — the tracker gates on viewport visibility, scroll velocity and active engagement, so a block that merely flashes past while you scroll does not count. Spend a few seconds on this paragraph and you'll see a text event appear once the dwell threshold is crossed.

Reads are reported with the time spent and whether they were partial. The same content embeds once on the server (keyed by a content hash) and is shared across every customer who saw it — so identical copy on ten landing pages is one embedding, attributed to ten passports at query time.

Images

Images marked with data-wb-image are tracked for viewport dwell. These ship with alt text, so the server uses that as the description instead of calling a vision model.

Video

The video below is marked with data-wb-video. The tracker records the disjoint intervals you actually watch and the completion percentage — not just "played". Note: on the server, a video the first viewer watches triggers transcription (Whisper + frame vision) of the watched portion, which calls OpenAI. Disable it by passing video:false in main.js if you'd rather not.

Verifying on the server

Copy the passport id from the header, then query the customer's timeline (needs the analytics bearer token). The reads you just performed appear as web exposures. See the README for the exact command.

Keep scrolling — the more you dwell, the more events fire. Reads that are still in progress when you leave the page are flushed via sendBeacon on unload, so even a quick visit is captured.
